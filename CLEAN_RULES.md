# 当前清洗规则

这份说明是给不写代码的人看的。

## 你的两个对外链接

- 主配置对外文件: `tvbox_config.json`
- 副配置对外文件: `jsm_backup.json`

它们给用户看到的是两个不同入口。

## 它们现在共用的一套规则

1. 先抓取上游原始配置
2. 保存一份上游留底
3. 再生成你自己的对外发布版本

## 具体规则

1. `sites`
   不是上游来什么就全发什么。
   会按你仓库当前已经保留下来的站点列表来生成。

2. `sites` 的顺序
   保持你现在发布文件里的顺序，不按上游顺序直接覆盖。

3. `sites` 的改名和改字段
   如果你现在发布文件里对某个站点改过名字、改过字段，工具会继续保留这些改动。

4. `lives`
   直播源直接跟着上游最新内容走，不做你自己的筛选顺序固化。

5. `spider`
   不再直接引用上游的 `spider.jar` 地址。
   工具会把它下载到你自己的 GitHub 仓库里，再自动生成你自己的 Raw 链接和最新 MD5。

6. `spider` 保护
   如果上游返回的不是一个真正的 jar 文件，而是网页、报错页、跳转页之类的假内容，工具不会覆盖你仓库里现有的 jar。

7. 主配置和副配置共用一个 `jar/spider.jar`
   所以只要这个 jar 变了，两个发布文件里的 spider MD5 会一起更新，保证一致。

8. 其他字段
   例如壁纸、解析、规则、DOH、IJK、广告等顶层内容，如果你当前发布文件里已经有人为调整，工具会尽量保留现有发布结果，不直接粗暴覆盖。

## 自动更新怎么工作的

不是上游一变化 GitHub 就能立刻知道。
因为上游不是你的仓库，所以只能“定时检查”。

现在工具已经配了 GitHub Actions：

- 每 6 小时自动检查一次上游
- 如果发现变化，就自动更新仓库
- 你的 TVBox 继续连你自己的 GitHub 地址，不需要改客户端

## 代理

你的本地工具默认会优先使用 Clash：

- `http://127.0.0.1:7890`

如果代理不通，会自动退回直连。

但 GitHub Actions 不会使用这个地址，因为 GitHub 服务器访问不到你本机的 `127.0.0.1:7890`。

## 你以后只需要做什么

1. 维护上游链接
   如果主链接拿不到，工具可以继续尝试候补链接。
2. 需要时手动点一次 GitHub Actions
3. 或者等它定时自动检查
